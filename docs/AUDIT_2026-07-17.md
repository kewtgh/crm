# Lumina Education CRM 最终架构与 UI/UX 审计

审计日期：2026-07-17
完成版本：0.7.0
范围：产品边界、认证、权限、数据、审批、销售、合同、产品、日历、通知、分析、国际化、UI/UX、测试与部署。

## 已核实的产品边界

CRM 登录身份仅限教育产品运营公司的员工：超级管理员、普通管理员、销售总监、销售经理、销售专员和销售支持。

- 无家长、学生、客户或联系人公开注册入口。
- Supabase Auth 匿名 signup 已关闭；删除应用页面并不是唯一防线。
- 客户、家长、学生和学校人员可以作为内部业务资料存在，但绝不是登录账号。
- 员工账号只由管理员创建、停用和分配角色；普通管理员不能创建管理员，管理员只能由超级管理员创建。
- 所有人登录都必须通过 Turnstile 服务端验证；失败在验证器旁提示并自动刷新。
- 新账号使用加密安全随机临时密码，邮件送达失败会回滚账号；首次登录必须修改密码。
- 超级管理员和管理员必须完成 TOTP MFA。AAL1 只允许完成改密/MFA，数据库权限助手同时阻止其读取 CRM 数据。
- 初始化邮箱和密码只是一次性引导凭据；脚本会创建真实 Auth 用户并移除 `ADMIN_PASSWORD`。
- 全仓未发现“小火种”“小火花”、导师、督导或 Little Spark 产品内容。

## 审计前主要问题及处理结果

| 问题 | 风险 | 处理结果 |
|---|---|---|
| 跨项目公开注册与身份流程 | 产生无业务归属的外部账号 | 路由、API、文案、环境变量和 Auth signup 均已移除/关闭；只保留后台创建 |
| 演示登录和固定凭据 | 配置错误可绕过真实认证 | 整个 demo 认证旁路已删除 |
| 过宽的 workspace 写权限 | 低权限员工可绕过 UI | 资源级 RLS、负责人/团队范围、受控 RPC 和 AAL2 已建立 |
| 审批只记录不执行 | “审批成功”与业务状态脱节 | 合同签署、业绩分配和导出生成均绑定幂等副作用 |
| 页面失败回退 fixture | 假数据被误认为真实数据 | 全部核心页改为真实错误/空状态；未完成模块明确停用 |
| 长列表浏览器假分页 | 数据量增大后性能与统计错误 | CRM、员工、合同、商机、审批、审计、通知和导出均服务端分页 |
| 假工作台/假角标/假搜索 | 运营判断失真 | 改为数据库聚合、真实通知和远程组织/联系人搜索 |
| 英文缺键回退中文 | 语言混杂 | 当前目录缺键直接暴露 key，并由目录一致性测试阻止缺键 |
| 合同/销售/产品只改本地状态 | 刷新后丢失 | 接入真实存储、版本、审批、业绩和付款口径 |
| 导出任务无执行器 | 审批后无法取得文件 | 私有 CSV、公式注入防护、签名下载、24 小时过期和失败诊断已完成 |
| 登录缺少人机验证 | 密码接口可被自动化撞库 | 全员 Turnstile、Siteverify、hostname 检查、单次 token 与登录限流已组合 |
| 管理员 MFA 只有设置页 | AAL1 会话仍可能访问后台 | 首次登录 MFA 设置、后续登录挑战、AAL2 Cookie 更新、页面/API/RLS 三层门禁已完成 |
| 邀请链接不符合账号策略 | 账号密码生命周期不明确 | 改为管理员创建、随机临时密码邮件、首次改密标记和失败回滚 |
| 审计函数不识别 `user_id` | 新员工偏好创建会回滚整个账号 | 迁移 018 恢复复合主键实体识别，并保留脱敏审计 |

## 已完成能力

- 现代统一设计系统：导航、搜索、分页、状态、表单、消息、抽屉、图标、颜色和响应式卡片。
- 登录就地错误、Turnstile 自动刷新、限流、Origin/Fetch Metadata、防点击劫持、CSP/HSTS 和员工状态校验。
- 完整账户设置：头像、双语姓名、称呼、用户名、内部 ID、邮箱、密码、语言、时区、通知、隐私、其他设备退出和 MFA。
- 中英文真正目录化；界面按当前语言显示，人员姓名是唯一固定双语例外；页头有语言切换。
- 管理运营门户、真实安全/MFA 指标、任务提醒、审批队列、员工创建/停用和审计分页。
- 销售总监/经理/专员与销售支持分支，目标分配、贡献归属、防重复、预测、漏斗和业绩分析。
- 四级关系目标：取得联系方式、共同用餐、可聊家长里短、可协助校内宣传；记录证据与覆盖率。
- 四个关系阶段和四个关单阶段各提供 4 条合规推进建议。
- 自定义产品、五个默认产品、多币种价格版本和未来生效日期。
- 合同草稿、签署审批、版本快照、续约父子关系、90/60/30/14/7 天提醒。
- 双月日历、用户时区、关联对象远程搜索、预约和站内/邮件提醒队列。
- 月/季/年消费看板、销售目标/实际/预测/漏斗和关系覆盖分析。
- 审批后的合同/业绩 CSV 生成、私有存储、短时签名下载和过期清理。

## 有意保持停用的模块

独立线索生命周期、批量导入、人工合并中心、数据质量规则中心、学生/家庭专属生命周期、升学进度和 AI 工作台尚未具备完整的持久化、权限、回滚和审计链。因此这些入口只显示“尚未启用”，不产生虚假成功。核心联系人查重、商机、客户组织和报告不受影响。基于“不矫枉过正”原则，本次没有为非核心入口匆忙制造第二套架构。

## 审计建议进入开发计划的核对

审计提出但不应在本轮仓促伪实现的能力，已经全部进入 `REMEDIATION_PLAN.md` 的后续路线图：客户 360 时间线、联系人同意/偏好与保留策略、报价到收退款对账闭环、汇率归一化、数据质量/导入/合并、Microsoft 365/Google Calendar/电子签/财务集成，以及带证据、置信度和人工确认的 Next Best Action。每项均列明优先级、依赖与验收门槛，不把“已进入计划”误写成“已经上线”。

## 最终风险与上线门禁

1. 正式 Supabase 控制台必须关闭公开 signup，并验证账号创建邮件域名。
2. 生产必须配置真实 Turnstile key/secret/hostname；不得沿用本地测试密钥。
3. 超级管理员和管理员必须使用 TOTP/AAL2，不存在开发环境放宽开关。
4. `SUPABASE_SERVICE_ROLE_KEY` 只存在服务器/worker 密钥管理中，绝不进入浏览器或仓库。
5. 账号创建邮件 webhook、outbox、提醒和导出 worker 需要调度、监控和失败告警；邮件系统不得记录临时密码。
6. 上线前完成 Chromium 1228 桌面、平板、375px、键盘焦点和横向溢出验收。
7. 执行备份恢复演练，并确认导出私有桶的过期清理按计划运行。
