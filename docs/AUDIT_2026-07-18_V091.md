# Lumina Education CRM 补充审计

审计日期：2026-07-18  
审计对象：v0.9.0 未提交工作树  
审计方法：计划—源码—数据库迁移—自动化测试—发布状态交叉复核

## 结论

v0.9.0 的基础构建和结构性测试通过，但“源码实现完成”的结论过早。当前仍存在两个发布阻断级安全问题、若干可靠性与数据完整性缺口，以及多项仅完成数据框架但尚未接入真实业务闭环的功能。

## P0：发布阻断

### 1. 停用员工可被旧 Auth 触发器重新激活

`handle_new_crm_membership()` 在 `auth.users.raw_app_meta_data` 更新后执行，并在 membership 冲突时无条件写入 `status='ACTIVE'`。员工停用流程先准备 `SUSPENDED` 状态，再更新 Auth metadata，因此会触发旧逻辑并恢复数据库 membership。

影响：

- 停用状态在 Auth 与业务 membership 间不一致。
- 已签发 JWT 仍可能继续满足数据库 membership 检查。
- 身份变更会被错误标记为已同步。

### 2. 运维任务重试未限制当前 workspace

`retry_operational_job()` 对通知、日历、生成任务和提醒的更新只按任务 UUID 定位，没有附加 `workspace_id=current_workspace_id()`。具备运维权限的用户在获知其他 workspace UUID 后可能重试不属于当前 workspace 的任务。

同时，人工重试没有写入审计事件。

## P1：安全、可靠性和数据完整性

### 3. 新增写接口未统一执行可信来源校验

Next Best Action、客户活动、运维重试、产品包/汇率、去重合并和续约 Playbook 等新增 POST 接口没有使用项目既有的 `mutationIsTrusted()`。产品目录和去重合并也没有保持敏感操作一致的 AAL2 门槛。

### 4. Webhook 缺少完整防重放协议

当前 HMAC 只覆盖请求体；事件 ID、事件类型和时间戳不在签名内容内。攻击者可复用合法请求体和签名，替换事件 ID 后重新提交。HTTP smoke 也没有验证签名失败、时间窗口和重放行为。

### 5. Worker 没有处理 claim 后崩溃

队列进入 `PROCESSING` 或 `SENDING` 后没有租约截止时间或超时回收规则。Worker 在 claim 后崩溃会产生永久卡死记录。Webhook 死信也没有出现在运维快照、人工重放和 readiness 中。

### 6. Readiness 与 SLA 指标不完整

Readiness 只统计部分队列，遗漏提醒、Webhook、导入和审批等关键工作流；运维中心显示积压和最老任务，但没有 SLA 目标、超限数量和卡死数量。

### 7. 重复记录合并可能丢失限制性 Consent

联系人 Consent 合并使用冲突忽略，然后删除来源记录。若目标记录为 `GRANTED`、来源记录为更新的 `REVOKED`，更严格的来源决定可能被丢弃。UI 还允许选择 SQL 未实际应用的字段，组织合并也没有迁移 Next Best Action。

## P1：功能闭环不足

### 8. 续约 Playbook 仍是手工写入表单

- 页面不会在编辑前加载已有 Playbook。
- 没有自动 90/60/30/14/7 天阶段、逾期标记和健康度。
- 合同 KPI 仍含固定“8 份履约正常”文案。
- 当前页面的部分生命周期视图只基于当前分页记录。

### 9. 产品包和汇率只完成目录框架

- 产品包没有版本和生效区间。
- 折扣上限没有接入报价校验。
- 产品包不能被报价或合同引用。
- 汇率快照没有关联报价、合同、付款或报表，无法证明历史金额锁定汇率。

### 10. 集成中心只完成状态表和通用入口

缺少连接/重新授权、配置更新、同步游标推进、同步执行和供应商业务适配器。电子签名和会计事件尚未驱动合同、付款或对账状态变化。

### 11. Next Best Action 缺少完整规则运营

当前规则需要手动生成，规则范围较少；没有“不适用原因”、生成批次、采纳率、拒绝原因分布和任务完成转化分析。

### 12. 分析建议未完整纳入原计划

原审计提出的留存、队列 SLA、预测准确率和规则建议采纳率分析没有形成完整实施项。

## P2：架构与 UI/UX

### 13. 统一 API 客户端没有被采用

`apiFetch()` 已存在，但应用仍保留大量直接 `fetch()`。结构化错误、会话刷新、超时/Abort 和错误消息行为不一致。

### 14. 身份生命周期和多租户模型仍有混合假设

员工创建与旧触发器存在固定 workspace UUID，而数据库又按多 workspace 设计。系统需要正式选择并约束单租户模式，或完成显式 workspace 选择和一致的员工 provisioning。

### 15. 部分错误被转换为空数据

产品、运维、导入和数据质量等页面的并行加载会把失败子请求转换为空数组，用户无法区分“没有记录”和“加载失败”。

### 16. 新增抽屉的键盘与焦点行为未统一

部分新抽屉缺少 Escape、初始焦点、焦点约束和关闭后的焦点恢复。浏览器运行时没有可用实例，桌面、平板、375px 和 axe 验收仍未执行。

## 测试覆盖缺口

现有 Node 契约主要检查源码结构；pgTAP 主要检查表、函数和权限存在性。以下关键场景缺少行为测试：

- 员工停用后 membership 与旧会话失效。
- 跨 workspace 人工重试拒绝。
- 新写接口的跨站请求拒绝。
- Webhook 时间窗口、头部篡改和重复提交。
- Worker claim 后超时回收。
- Consent 冲突合并与组织关联完整性。
- 产品包折扣边界、汇率锁定和续约窗口。
- NBA 采纳、拒绝和任务完成转化统计。

## 发布状态

- 当前工作树仍未提交。
- Sites 项目存在，但没有保存版本、运行时环境和部署。
- 生产 Worker、外部供应商凭据和 readiness 尚未验证。
- 浏览器视觉与无障碍验收仍是发布门禁。

## 修复原则

1. 所有安全边界必须在数据库或服务端执行，不能只依赖 UI。
2. 所有人工敏感操作必须带 workspace 条件、AAL2、可信来源校验和审计。
3. 所有异步任务必须具有租约、超时恢复、幂等、退避、死信和重放闭环。
4. 所有新业务表必须真正被报价、合同、任务或分析流程引用，避免“只建表不落地”。
5. 测试必须覆盖失败路径和跨租户行为，而不仅检查源码字符串。

## v1.0.0 处理结果

截至 2026-07-18，本审计列出的 P0、P1、P2 仓库内可执行事项均已关闭：

- 身份停用、两步建档、显式 workspace 和修复队列已闭环；
- 所有敏感写入具备来源、AAL2、租户和审计边界；
- Webhook 使用完整 canonical envelope、5 分钟窗口和原子幂等摄取；
- 六类 Worker 使用租约/fencing token、超时回收、退避、死信、重放、SLA 与心跳；
- Consent-safe 合并、自动续约、版本产品包、汇率锁定、provider-confirmed 集成同步、NBA 评估与经营分析均已接入真实业务流程；
- 统一 API 客户端、共享可访问 Drawer、错误/空状态和移动字段标签已完成。
