# Lumina Education CRM v2.1.1 综合审计

- 审计日期：2026-07-20
- 审计对象：v2.1.1 当前工作树
- 审计范围：发布配置、Worker、数据库状态机、RLS/API、报表与导出、运营恢复、
  UI/UX、响应式、可访问性证据、架构、测试、版本与发布材料
- 基线结果：TypeScript、ESLint、production build、25 项 Node 测试、275 项 pgTAP、
  PostgreSQL schema lint 与 npm dependency audit 全部通过

## 结论

工程基础健康，但自动化门禁没有覆盖全部跨层业务闭环。当前版本不应在修复下列
发布阻断和高优先级问题前被标记为全面 production-ready。

## P0：发布阻断

### WORKER-01 正式 Worker 的邮件变量名不一致

`.github/workflows/production-workers.yml` 提供 `EMAIL_DELIVERY_URL` 与
`EMAIL_DELIVERY_TOKEN`，通知和日历 Worker 实际要求
`EMAIL_DELIVERY_WEBHOOK_URL` 与 `EMAIL_DELIVERY_WEBHOOK_TOKEN`。正式统一 Worker
周期会在邮件相关步骤失败。

### PRIVACY-01 隐私请求只有审批状态，没有实际执行闭环

`manage_privacy_request` 对 EXPORT/DELETION 执行双人审批，但最终只完成 CRM 任务并把
请求更新为 `FULFILLED`。ACCESS、EXPORT、CORRECTION、RESTRICTION、DELETION 均缺少
类型专属执行器、数据范围快照、限制落实、删除/匿名化清单、法律保留例外、产物校验和与
不可抵赖执行凭证。

## P1：业务正确性与运营韧性

### EXPORT-01 绩效导出错误合并多币种

绩效导出读取目标和付款币种，却按人员直接相加并输出无币种的单一数字。跨币种数据会被
错误汇总。

### EXPORT-02 导出存在静默截断

通用 CRM 导出最多读取 10,000 条后仍会成功；绩效和营销导出依赖单次 REST/RPC 返回，
没有预期行数、实际行数、截断检测或完整性证明。

### CURRENCY-01 销售报表口径不一致

`sales_performance_report` 选择一个币种统计金额与 funnel，但同一成员的商机数量不按币种
过滤。列表、数量和金额可能表达不同范围。

### OPS-01 可选集成阻塞核心 readiness

统一 Worker 无条件运行 Webhook 与 integration sync，release readiness 也强制要求所有
外部集成 secrets。没有启用第三方集成的核心部署无法健康，与“按启用能力配置”的产品边界
不一致。

### OPS-02 集成同步失败无法从运营中心重试

`INTEGRATION_SYNC` 已进入心跳和队列指标，但不在 retryable jobs 页面及
`retry_operational_job` 的分支中。

### AUTHZ-01 合同能力在 UI/API/数据库间漂移

合同页面对所有可访问用户显示创建和续约动作，API 只要求登录，数据库最终拒绝
`SALES_SUPPORT`。这会形成填完表单后才失败的权限体验，且权限定义无法集中审计。

## P2：UI、发布证据与可维护性

- 移动端全局搜索占位文案被截断，空状态卡片和桌面空列表保留过多无效空间。
- 隐私高风险操作使用常驻宽表单，缺少按类型解释、执行范围预览和分步骤确认。
- `/api/health?mode=ready` 的 queue check 只检查 failed jobs，不检查 stuck jobs。
- Open Graph 仍引用包含 `v1.2.0` 的旧图片；部署说明、实现状态和 v2.1.1 版本漂移。
- Chromium QA 报告没有 Git SHA、应用版本、迁移 head 或构建哈希；历史截图显示 v2.1.0，
  不能归因到当前 v2.1.1。
- 浏览器矩阵未覆盖合同、日历、消息、设置、运营中心等高风险路径。
- `components/v200-workspaces.tsx`、`app/globals.css` 和若干客户端页面承担过多领域职责。
- Node 契约偏重源码结构断言，需要补充真实状态转换、队列和导出行为测试。

## 建议纳入的产品扩展

1. 事件驱动工作流自动化：触发器、条件、动作、预览、幂等运行与审计。
2. 招生旅程和渠道归因：活动、UTM/触点、阶段漏斗与 ROI。
3. 家长/学生门户：独立于员工身份的邀请、同意、资料查看与更新请求。
4. 统一沟通收件箱：线程、人工记录、经同意渠道发送、投递状态和供应商适配边界。
5. 多币种经营报表：基础币种、汇率快照、原币与折算币并列呈现。
6. 数据质量可观测性：规则、问题队列、趋势、负责人和批量修复。
7. 电子签、付款与会计连接器：显式启用、最小权限、重放保护、对账和审计。

## 正向发现

- 认证、可信设备、MFA、Turnstile、Origin 校验、持久限流和 staff-only 边界较完整。
- 多租户表普遍使用 RLS，审批、导出和运营任务已有 maker-checker、lease 与审计基础。
- 当前依赖审计没有已知漏洞，数据库从行为层已有 275 项 pgTAP 覆盖。
- 历史 Chromium 证据显示主要现有页面的响应式、标签、焦点、溢出和对比度基础良好。

## v2.2.0 实施后的第二轮遗漏复审

首次实现后再次逐项对照“完成定义”，发现若干功能已有页面或表，但还未形成完整闭环：

- 自动化缺少无副作用预览、规则版本和失败运行重试。
- 门户邀请未在服务端证明收件邮箱属于目标家庭；授权前仍可取得资料，提交请求也可绕过
  UI 的同意步骤；批准请求没有把受支持字段应用到主档并记录应用明细。
- 沟通消息缺少幂等键、人工来信、全文搜索、尝试次数和失败重试。
- 数据质量只有趋势，缺少规则启停/严重度、负责人及监护关系、过期同意、汇率、归因和
  重复联系人规则。
- 增长页缺少按币种管线/赢单、入读、ROI 和 Dashboard 跳转指标。
- 移动端全局搜索仍是被压缩的桌面输入框，没有完整焦点管理和结果对话框。
- 连接器虽有 replay protection，但缺少可读、不可变且跨工作区安全的对账 receipt。

以上源码缺口已由迁移 `202607200051_expansion_completion.sql` 及对应 API/UI/双语/测试补齐。
复审还发现并修复两类静默幂等风险：同一 communication key 对应不同正文，以及同一 connector
event ID 对应不同 payload；两者现在都会明确拒绝。

后续验证已在隔离本地数据库实际应用至 `052`，schema lint 为 0，9 个 pgTAP 文件共
433 项全部通过。真实 Worker 验证又发现并修复 `service_role` 缺少日历/隐私导出来源读取
权限的问题，同时将分页测试改为不受既有失败队列污染的事务隔离 fixture。浏览器矩阵尚未执行；开发环境已安装固定的
`ms-playwright/chromium-1228`，后续必须直接使用该运行时，不能以缺少 in-app Browser
会话为由停止。扩展浏览器矩阵和 Sites 发布仍未记为通过，精确状态见整改计划和实现状态文档。
