# Lumina Education CRM v2.1.0 综合审计

- 审计日期：2026-07-20
- 审计基线：`c1f9cdf`
- 审计范围：产品范围、数据库与 RLS、API、业务状态机、前端交互、双语、移动端、
  可访问性、构建、测试、Sites 托管状态
- 基线验证：TypeScript、ESLint、production build、23 条 Node 契约、222 条 pgTAP
  与 schema lint 均通过

## 结论

v2.0.0 已具备较强的认证、审计、RLS、异步任务、导入导出和响应式基础，但“自动化
门禁通过”不等于业务闭环完整。源码复审确认 1 个发布级业务阻断、5 个高优先级业务/
权限缺口和若干体验问题。此前“没有已知 P0/P1/P2 缺陷”的结论已不再成立。

## P0：业务阻断

### PROG-01 升年级批次无法应用

预览时保存随机 `idempotency_key`，UI 应用时却把批次 UUID 当作同一个 key 提交；
数据库函数要求二者相等，因此正常创建的预览批次必然报
`progression_idempotency_mismatch`。现有 pgTAP 只断言函数源码包含幂等校验，没有执行
“预览 → 应用”的真实行为，产生假阳性。

## P1：业务正确性、权限与产品范围

### PROG-02 当前年级算法不安全且与产品文案不符

系统直接对年级字符串中的首个数字加一。非数字年级可能令整批预览失败；系统没有可配置
年级规则、逐项预览/排除、例外原因或回滚入口。应用时也没有关闭旧 `CURRENT` 学籍并写入
新学年记录，导致学生主档和学术时间线不一致。

### LEAD-01 家庭 Lead 不能转化

UI 和数据库都只允许学校 Lead 转商机；`opportunities.organization_id` 又是必填。
产品虽然可创建家庭 Lead 并显示“学校与家庭独立 Pipeline”，但家庭 Pipeline 没有成交
入口，属于业务声明与执行能力不一致。

### AUTHZ-01 Lead capability 与 RLS 漂移

`SALES_SUPPORT` 在共享 capability 中拥有 `leads.view`，导航和页面会开放；数据库策略却
明确排除该角色。最终表现是用户看到合法页面但数据被静默过滤为空，而不是清晰的权限边界。

### EDU-01 家庭成员与监护关系只读

数据表、详情读取和文案已存在，但 UI/API 不能添加、修改或移除家庭成员和学生监护人，
也没有“唯一主要联系人”等数据约束。关系模型尚未形成可执行维护闭环。

### ERR-01 数据库业务错误被折叠

PL/pgSQL 业务规则通过 `raise exception 'education_version_conflict'` 等方式返回，但
PostgREST code 通常只有 `P0001`。服务端丢弃安全的业务 message，客户端无法区分并发冲突、
状态错误和资格错误，统一错误呈现器实际得不到所需 code。

## P2：发现性、运营与 UI/UX

### SEARCH-01 v2 核心对象未进入全局搜索

顶部搜索覆盖学校、联系人、任务、商机、合同、报价和产品，但不搜索学生、家庭或 Lead。
用户无法从全局入口直达新增的教育和获客对象。

### DASH-01 工作台没有教育与获客信号

Dashboard 只展示任务、审批、合同、付款、产品和通知；PRD 要求的新 Lead、活跃学生、
待确认升年级批次等关键运营信号缺失。

### AI-01 建议运行会重复生成

同一用户可反复为同一合同生成多条仍有效的 OPEN 建议；已接受/拒绝的建议又被 UI 隐藏，
但仍计入分页总数，可能出现“页面为空、总数非零”的认知冲突。

### UI-01 导入文件控件未完整本地化

移动端实机证据显示原生文件输入仍出现英文 `Choose File / No file chosen`，中文页面的
主要操作未本地化；标题写“CSV”而实际支持 CSV/XLSX。

### UI-02 空数据移动布局信息密度过低

学生和 Lead 等页面在空数据时仍保留桌面级大面积列表高度，搜索、空状态和分页被拉成
多个视觉区块；移动端需要更紧凑的空状态和更明确的首个操作。

### RES-01 若干 v2 列表重载缺少一致失败态

Lead、隐私请求、AI 建议的部分重载函数未捕获失败；事件触发的 rejected Promise 可能只
进入 console，而页面保留旧数据且不给用户重试说明。

## 补充认证审计（用户边界确认后）

### AUTH-01 强制 MFA 角色范围过宽

应用层和 v2.0 数据库 helper 把销售总监、销售经理与管理员一起放入全局 AAL2 登录门槛，
这与最终业务规则“只有超级管理员和管理员必须启用 MFA”不一致。销售角色不应因未启用
MFA 而失去整个 CRM 工作区；其他员工仍应保留自愿启用 TOTP 的能力。

### AUTH-02 新设备验证与可信设备缺失

原登录只接受邮箱/密码。`remember` 仅延长 refresh cookie，没有为未启用 MFA 的普通员工
提供首次/新设备邮箱验证码，也没有服务端可信设备记录、过期、单独撤销或安全变更失效。
Turnstile 已覆盖密码登录，但认证链路仍未满足“账户名登录、首次邮箱验证、后续可信设备仅
密码”的完整规则。

### AUTH-03 管理员保存已验证浏览器的语义不完整

密码登录时选择保持登录，但 MFA verify 会用 session cookie 覆盖持久 refresh cookie，
导致完成 AAL2 后无法按预期保存已验证浏览器会话。需要把登录时的记住选择安全传递到
TOTP verify，并保持 Supabase AAL2 refresh session，而不是伪造 AAL2 token。

## 架构与质量发现

- `components/v200-workspaces.tsx` 同时承载五个业务工作台，修改风险和回归面过大；本轮先
  抽取共享行为和明确类型，不进行无收益的大规模视觉重写。
- 现有 Node 测试仍以源码契约为主；数据库测试应增加“真实执行行为”，特别是升年级、
  家庭 Lead 转化、关系维护和低权限读取。
- Sites 项目为 owner-only，但生产环境变量仍为 0 条，当前线上版本是旧提交。没有真实
  Supabase、Turnstile、限流和 worker 配置时不得把新版本部署为可用生产系统。

## 正向发现

- 认证 cookie、首次登录改密、MFA/AAL2、Origin 校验、持久限流和服务端角色边界完整。
- 租户表普遍启用 RLS，敏感导出、隐私请求和审批具备审计与 maker-checker 基础。
- production build、schema lint 和已有测试在基线均通过，没有依赖漏洞证据或脏工作树。
- 1440/1024/375 历史 Chromium 证据显示整体视觉、横向溢出、标签、焦点与对比度基础良好。

## 本轮纳入 v2.1.0 的完整建议

1. 修复升年级幂等阻断，加入可配置年级规则、逐项预览/选择、例外 HOLD、学术快照和批次取消。
2. 扩展商机主体为学校或家庭，打通家庭 Lead 转化并保留 pipeline/转化证据。
3. 对齐 Lead capability/RLS，并增加可执行权限测试。
4. 补齐家庭成员和学生监护关系的添加、更新、移除与主要联系人约束。
5. 安全恢复数据库业务错误 code，使 UI 能提示并发冲突、状态约束和缺失配置。
6. 将学生、家庭、Lead 纳入全局搜索，并在 Dashboard 增加教育/获客指标。
7. 对规则建议做“同主体同用户仅一个有效 OPEN 建议”的去重，并提供决定历史。
8. 本地化文件选择控件，压缩移动空状态，统一 v2 重载失败反馈。
9. 以行为 pgTAP、Node 契约、生产构建、HTTP/Chromium 门禁和最终遗漏复审作为完成标准。
10. 仅管理员强制 MFA；其他员工可选 TOTP。未启用 MFA 的普通员工在新设备执行邮箱 OTP，
    可保存 30 天可信设备；密码登录始终由 Turnstile 与持久限流保护。

## 执行后复审结论

- P0/P1/P2 与补充认证发现均已完成整改；对应实现见
  `REMEDIATION_AND_PRODUCT_PLAN_V2.1.0.md` 的已勾选清单。
- 从空库应用至 `202607200042` 成功，schema lint 0 findings，275 项 pgTAP 全部通过。
- 真实认证门禁已验证 Turnstile、账户名/密码、邮件 6 位 OTP、首次改密、可信设备复用；
  可信设备只保存 HMAC、使用 HttpOnly cookie、30 天过期并支持用户撤销与审计。
- Chromium 149 对 27 个页面/视口完成检查，并执行全局搜索、家庭成员、学生监护、学年
  晋级和家庭 Lead 转化。复审期间发现的学年晋级 PostgREST 关系歧义也已修复并纳入失败门禁。
- 生产 Sites 当前仍无运行时环境变量；正式 Supabase、Turnstile、OTP 邮件模板、worker
  与 readiness 200 是发布外部门槛，不属于可以在仓库中伪造完成的事项。
