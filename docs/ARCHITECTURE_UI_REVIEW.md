# 架构与 UI/UX 复检报告

版本：0.2.0
复检日期：2026-07-16

## 结论

当前实现是一套完成度较高的现代化 CRM 交互基线，信息架构、双语姓名、独立登录/注册、管理门户、通用搜索与分页组件都已成形；但它仍不是可承载真实客户数据的生产 CRM。最主要的原因不是页面数量，而是业务数据仍来自前端 fixture，写操作和个人设置多为提示反馈，没有工作区数据层、RLS、审计事务和服务端分页。

本轮已修正可以在现有边界内安全完成的问题：

- 角色只信任 Supabase `app_metadata`，不再从用户可编辑的 `user_metadata` 授权。
- `/admin/**` 增加服务端管理员校验，非管理员导航不再暴露管理入口。
- 未获工作人员角色的家长注册账号不能进入 CRM 工作区。
- 增加访问令牌刷新路由；过期会话优先尝试使用 refresh token，而不是直接丢回登录页。
- 补齐忘记密码接口和新密码页面，错误留在对应表单内，URL 不携带错误原因。
- 生产登录页不再显示演示账号；Turnstile 增加 action/hostname 校验和失败刷新。
- 可搜索下拉增加键盘方向键、Enter、Escape、焦点返回和 ARIA 关联。
- 提升小字号表格、状态和移动端触控目标的可读性。

## 按风险排序的剩余问题

### P0：上线阻断

1. **业务数据没有生产数据层**：学校、人员、学生、家庭、商机、任务、验证队列和导师列表都来自 `lib/crm-data.ts` 或组件内数组。创建、编辑、导出、设置保存、MFA 和会话管理没有真实持久化。
2. **没有工作区隔离与 RLS**：运行时没有应用数据库迁移、成员关系、workspace_id 约束或 Supabase RLS。即使 UI 有角色，仍不能安全承载多学校/多团队数据。
3. **长列表仍是客户端分页**：当前组件先取得全部数组，再搜索和 `slice`。真实数据必须采用服务端 `query + cursor/page + pageSize + total`，并设置最大页大小。
4. **权限模型只完成管理员边界**：管理员路由已封闭，但导师、督导、销售对各业务对象的读写范围仍需权限矩阵与数据库策略共同实现。
5. **设置与安全是界面原型**：头像、邮箱、密码、通知、隐私、其他设备退出和 MFA 尚未接入 Supabase/Storage/后端服务。不能以页面存在等同功能完成。
6. **缺少可执行的浏览器回归**：本次复检时应用内浏览器运行时未发现 Chromium 实例，因此没有完成真实点击、截图和移动端视口验证。现有自动测试主要验证构建和渲染标记，覆盖不足。

### P1：进入真实开发后优先处理

1. **国际化没有统一消息层**：资料字段支持中英姓名，但界面语言选择器没有驱动全站文案、日期、时区和通知模板。应引入消息目录和 locale 路由/会话偏好，不要继续散落硬编码双语字符串。
2. **组件文件和全局 CSS 偏大**：`operations-pages.tsx`、`settings-page.tsx`、`admin-pages.tsx` 与 `app/globals.css` 承载过多页面。建议按业务域拆分，不需要改成复杂 monorepo，也不必为了“纯洁架构”过度抽象。
3. **假性可操作控件较多**：筛选、导出、保存视图、看板添加、AI 审核、通知已读等按钮会给出可用暗示，却没有后端行为。接入前应禁用并标注，或完成最小闭环。
4. **移动端表格依赖横向滚动**：管理表格宽度超过 1000px。桌面表格可保留，但移动端应切换成摘要卡片或“关键列 + 详情抽屉”，而不是仅缩小字号。
5. **异步状态体系不完整**：真实请求接入后需要统一 loading、empty、error、retry、offline 和权限拒绝状态，并增加路由级 `loading.tsx`、`error.tsx`。
6. **抽屉/弹层焦点管理仍需加强**：可搜索下拉已修正基础键盘能力；记录抽屉、通知和个人菜单还需要焦点圈定、Escape 关闭及关闭后焦点恢复。

### P2：视觉优化

1. 现有配色、留白、卡片层级和导航结构总体一致，问题主要是信息密度过高而不是风格落后。
2. 本轮只提升了关键小字和触控基线。后续应使用真实长文本、中文/英文混排和 320/375/768px 视口做视觉回归，再局部调整，避免全局放大造成布局反向失衡。
3. 状态色已有文字标签，方向正确；真实数据接入后需要检查高对比模式、焦点可见性和图表的非颜色表达。

## 建议的目标边界

```text
Browser
  -> Vinext/Next BFF（会话、校验、权限、限流）
      -> Supabase Auth（身份、MFA）
      -> PostgreSQL + RLS（workspace 数据、审计、服务端分页）
      -> Storage（头像、附件）
      -> Temporal Worker（导入、升年级、批处理）
```

列表查询统一返回 `{ items, total, page/cursor, pageSize }`；所有写操作在服务端完成 schema 校验、工作区授权、重复检查和审计记录。角色来自服务器维护的成员关系或 `app_metadata`，用户资料只保存显示信息，绝不能参与授权判定。

## 克制的执行顺序

1. 先建立 workspace、membership、核心实体和审计表，并完成 RLS 自动化测试。
2. 选择一个垂直切片（建议“学校 + 联系人 + 任务”）接入真实 CRUD、查重和服务端分页。
3. 接入真实个人资料、密码、设备会话和 MFA，再开放设置页操作。
4. 建立 locale 消息层并完成中英切换，不同时重写所有页面布局。
5. 使用指定 Chromium 1228 完成登录、注册、Turnstile 失败、权限、移动端、长列表与设置回归。
6. 垂直切片稳定后，再逐域替换其余 fixture 页面。

这一路径把安全和数据边界放在最前面，但避免一次性重写全部 UI。
