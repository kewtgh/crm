# 架构、UI/UX 与指令中断审计

版本：0.4.1
审计日期：2026-07-16

## 审计结论

本轮在 v0.2.0 基线上完成了用户追加的日历、销售、合同、产品、消费分析、关系推进、关单建议、真实国际化和独立身份模型，并重新检查了多次追加指令是否让较早任务中断。已发现并补齐的中断缺口包括：旧后台/运营/设置页尚未接入语言目录、销售和首页残留硬编码界面文案、产品自定义计费保存了翻译文本、旧架构报告仍把国际化列为未完成。

本次又将实现与原始压缩包中的 `01_PRD.md`、`02_PAGE_INFORMATION_ARCHITECTURE.md` 和数据模型逐项比对。基于新增业务要求，权限模型明确为超级管理员、普通管理员，以及销售总监、销售经理、销售专员和独立的销售支持支线；没有重新引入来源不明的其他项目角色。学生、家庭、监护关系和升年级在原始 PRD 中有明确领域定义，因此继续保留。

当前版本适合作为受保护的 UI、认证和交互验收环境。它不是可承载真实学生、家庭和客户资料的生产 CRM；业务记录与大多数写操作仍是 fixture 或会话内原型。界面不会把这些原型误写成生产完成状态。

v0.4.1 又通过实际 Chromium 1228 回归发现并修复了此前静态审查遗漏的问题：客户端组件误引用 `next/headers` 所在认证模块、`AppShell` 热更新后用户 Context 分裂、认证表单在水合前可能以 GET 降级、正式登录页暴露演示凭据且隐藏找回密码入口、页面标题重复品牌后缀、英文切换后 metadata 不刷新、设置/认证/运营页残留英文 eyebrow，以及移动端业绩分配百分比显示过长。

## 仍是上线阻断项（P0）

1. **业务数据层与工作区隔离**：建立 workspace、membership、核心业务表、审计表、附件存储及 RLS；当前 fixture 不得导入真实敏感资料。
2. **服务端分页与查重**：真实列表统一返回 `{ items, total, page/cursor, pageSize }`，设置最大页大小；查重在服务端事务内重验。
3. **完整权限矩阵**：管理员和销售层级已进入认证白名单，审批/分配已有 RLS；学校、家庭、学生及其他业务对象仍需 workspace 范围策略。
4. **真实设置和安全服务**：头像、通知、隐私导出、密码、其他设备退出、MFA 状态及恢复代码仍是明确标注的验收界面。
5. **后台任务**：日程通知、合同续约、目标版本、导入和批处理需要幂等 worker、重试、死信和审计。
6. **业务动作闭环**：合同签订/导出、业绩汇总与分配的审批界面已经可验收，数据库迁移也已提供，但页面仍未通过 BFF 事务写入审批表；刷新后状态会还原，不能作为正式审批凭证。

## Chromium 1228 实测结果

- 使用 `ms-playwright/chromium-1228`、1440px 桌面与 375×812 手机视口完成登录、注册、工作台、审批、业绩分配、合同、销售分析、双月日历和设置页回归。
- 所测页面 `document.scrollWidth` 均未超过视口；手机端检测到的负坐标元素是关闭状态的离屏侧栏，不会造成横向滚动。
- 错误凭据返回 401，但错误文字位于登录表单内，URL 保持 `/login`；审批驳回原因和分配成员错误都显示在当前表单附近。
- 中英文切换后 H1、eyebrow、ARIA 文案和浏览器标题同步切换；姓名继续按“中文 / English”双语显示这一明确例外。
- 修复后未再出现客户端 `async_hooks` 或用户 Context 运行时错误。控制台仅保留测试刻意触发的登录 401，以及本地无生产域名时 Turnstile 外部脚本受 COEP 拦截；后者仍须在正式域名和生产密钥下验收自动刷新。

## 下一轮优先级（非本次补丁伪装完成）

1. **P0：真实垂直切片**——先完成“学校 + 联系人 + 任务”的数据库、BFF、RLS、审计、查重和服务端分页，再依次替换 fixture。
2. **P0：审批/分配持久化**——用服务端事务连接既有 migration，加入幂等键、状态机、审批人隔离、通知与审计，不允许客户端直接改最终状态。
3. **P0：设置与安全服务**——头像、通知、邮箱/密码、其他设备退出和 MFA 仍需接 Supabase Auth、Storage 与审计；当前仅是明确标注的验收界面。
4. **P1：空操作清单**——旧模块表格的状态/负责人筛选、排序、批量分配/标签，以及部分门户快捷按钮仍是视觉占位。接数据服务时应统一接 action registry；在此之前应禁用并说明，而不是给出虚假成功提示。
5. **P1：真实数据压力与可访问性**——用长姓名、长校名、多币种、大金额、空数据、1000+ 条分页和键盘/读屏流程复测；当前 fixture 无法覆盖这些边界。
6. **P1：提醒 Worker**——预约、合同续约、导入和通知需要幂等调度、重试、死信队列、时区策略和可观测性。

## 克制的下一阶段架构

```text
Browser
  -> Vinext/Next BFF（会话、校验、权限、限流）
      -> Supabase Auth（身份、MFA）
      -> PostgreSQL + RLS（workspace、业务数据、审计、服务端分页）
      -> Storage（头像、附件）
      -> Worker（提醒、导入、续约、批处理）
```

不建议为了“现代化”改成复杂微服务或 monorepo。先选择“学校 + 联系人 + 任务”作为真实垂直切片，完成 CRUD、查重、RLS、审计和分页，再按域替换 fixture；这与当前风险和修改幅度相匹配。

## UI/UX 复检

- 导航层级、卡片、颜色、图标、搜索、分页、消息和表单反馈已统一；状态不只依赖颜色。
- 手机断点已覆盖日历、销售 Playbook、审批、业绩分配、合同、产品和分析页；本轮所测核心页面均无文档级横向滚动，宽表在手机端转为摘要布局。
- 登录、注册、密码和设置错误在对应表单附近；错误原因不进入 URL。
- 人员姓名保持“中文 / English”同时显示；学校、产品、菜单和说明只随当前语言显示。
- 抽屉支持 Escape 和焦点返回的页面已覆盖新增核心流程；其他旧菜单的焦点圈定、表格排序状态提示和批量操作键盘流仍需专项可访问性回归。
