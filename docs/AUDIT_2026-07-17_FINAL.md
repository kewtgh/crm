# Lumina Education CRM 综合审计报告

审计日期：2026-07-17  
审计基线：v0.8.0 / `e7a6a6a`  
审计范围：产品边界、数据库权限、认证、审批、报价、合同、收退款、导入、数据质量、后台任务、API、UI/UX、国际化、无障碍、测试、部署和运维。

## 结论

项目的类型检查、Lint、生产构建和依赖安全检查均通过，42 张业务表均显式启用 RLS。但 v0.8.0 不能作为生产发布版本：旧退款 RPC 仍可绕过新审批流，多个 Security Definer RPC 将角色或对象级权限留给 API 层处理，报价关联缺少同 workspace 校验，API 未登录响应不一致，账号停用与会话撤销也没有形成原子闭环。

UI 层还存在固定关系分数、固定“Verified / MFA enabled”、失效快捷键、设置加载竞态、移动表格缺字段标签、错误状态不完整和已移除产品范围文案残留。现有 Node 测试主要是源码契约检查，不能证明业务状态机、API、键盘操作和跨租户授权正确。

## P0：发布阻断

### 1. 旧退款接口绕过审批

`202607170011_sales_intelligence.sql` 中的 `refund_payment` 直接把付款标记为退款，并继续向 `authenticated` 开放。v0.8.0 新增的 `request_refund` / `complete_refund` 要求申请、审批、凭证和对账刷新，但旧函数没有被撤销。调用者可以绕过 maker/checker、退款台账、`refunded_amount`、应收计划和对账差异。

验收要求：

- 前向迁移撤销并删除旧退款函数。
- 新退款函数校验角色、付款对象可见范围、金额、付款状态和 workspace。
- 数据库负向测试证明旧签名不可执行、跨团队付款不可退款、自审不可通过。

### 2. 数据库权限依赖 API/UI

以下能力在 API route 中有限制，但数据库函数本身没有完整限制：

- `create_approval` 没有对所有业务对象执行 owner/team 写权限检查。
- `request_refund` 没有校验付款所属合同的对象级访问权。
- `request_marketing_contact_export` 没有限制为销售领导或管理员。
- `resolve_data_quality_issue` 没有限制为销售领导或管理员。
- `approval_requests` 和 `approval_actions` 仍允许 authenticated 直接 INSERT。

Supabase 访问令牌允许浏览器绕过应用 API 直接调用 RPC 或表接口，因此数据库必须是最终授权边界。

验收要求：

- 撤销审批表直接 INSERT。
- 所有业务 RPC 同时验证身份、角色、workspace、owner/team、状态和输入。
- 关键写入只允许受控 RPC 或 service role。

### 3. 报价存在跨 workspace 关联风险

`create_quote` 校验了组织，却没有验证 opportunity、product 是否属于相同 workspace、商机是否属于该组织、产品是否启用。Security Definer 会绕过普通 RLS，简单 UUID 外键也不能防止跨 workspace 引用。

验收要求：

- RPC 内验证机会、产品、组织和 workspace 的一致性。
- 为新模型增加组合唯一键与组合外键或等效约束触发器。
- 负向测试覆盖跨 workspace UUID。

## P1：安全、架构与业务一致性

### 4. API 身份认证响应不一致

页面和 API 共用会 `redirect()` 的 `requireUser` / `requireRole`。运行时确认：

- `/api/crm/schools` 未登录返回 307 和 HTML 登录页。
- `/api/admin/users` 捕获重定向异常后返回 500。

这会让前端 fetch 在会话过期时发生 JSON 解析错误，也会把正常 401 计入服务端故障。

验收要求：

- 页面 guard 继续重定向。
- API guard 返回统一 JSON 401/403/MFA_REQUIRED。
- 客户端请求层统一解析错误、处理会话过期和重试。

### 5. 管理员账号更新非事务化

账号变更依次写 Auth metadata、workspace membership 和 team membership。中途失败会造成身份声明、数据库角色和团队状态分裂。停用文案宣称已撤销访问，但更新流程没有显式执行全局 session revoke。

验收要求：

- 数据库角色与状态变更使用事务 RPC。
- Auth metadata 同步使用补偿或 outbox。
- 停用明确撤销用户会话；失败可见、可重试、可审计。

### 6. 登录限流只存在于进程内

`lib/login-rate-limit.ts` 使用模块级 `Map`。它无法跨 Cloudflare isolate、实例或重启共享。

验收要求：

- 数据库提供原子限流 RPC，生产环境不依赖本地内存。
- 同时按账号哈希和来源哈希计数，不保存明文邮箱/IP。
- Turnstile、限流和失败审计保持组合防护。

### 7. 后台任务缺少统一运维闭环

通知、提醒、日历和导出分别由手工 Node 脚本处理。缺少统一的任务新鲜度、队列积压、重试、死信、卡死恢复和调度健康视图。

验收要求：

- 建立 worker heartbeat、队列指标和受控重试 RPC。
- 健康检查区分 liveness/readiness，并暴露数据库、REST、队列和 worker 新鲜度。
- 管理员运维页可查看积压、失败和最近运行时间。

## P1：UI/UX 与内容完整性

### 8. 固定指标和固定安全状态

- 侧边栏固定展示关系健康度 87% 和本周提升 3.2%。
- 用户菜单固定展示 Verified 和 MFA enabled。
- 设置页固定展示 Active · Verified。

这些文案没有来自真实业务或身份数据，违反“不得伪造成功和指标”的产品原则。

验收要求：接入真实指标和状态；无法证明时显示未配置、未知或行动入口。

### 9. 设置页加载竞态

设置页先用 fallback 渲染可编辑表单，再异步加载真实数据。用户可以在加载完成前保存默认值；加载失败后表单仍可写，可能覆盖真实设置。邮箱被修改后仍显示已验证。

验收要求：只有加载成功才允许保存；失败时保留只读状态和重试；邮箱显示当前已验证值和待验证新值。

### 10. 搜索、导航与弹层无障碍

- 搜索框显示 `⌘ K`，但没有快捷键处理，也不适配 Windows。
- 搜索没有 loading/error/listbox/active descendant/Escape/焦点恢复。
- 通知、用户菜单和分组导航缺少 `aria-expanded`、外部点击和 Escape。
- 非管理员会在“管理”分组中看到设置入口，信息架构含混。

验收要求：实现真实快捷键、完整 combobox 语义、键盘导航、弹层关闭和按角色分组。

### 11. 表格和网络错误状态不完整

- 通用表格网络异常时可能保留旧数据或产生未处理 Promise。
- 固定每页 5 条，状态筛选是循环按钮，批量选择只有“取消”。
- URL 数字参数允许 NaN，Malformed query 可能形成 500。
- 移动卡片移除表头后没有字段标签。

验收要求：共享查询解析、加载/失败/重试、可发现筛选、真实批量动作或移除选择、响应式字段标签。

### 12. 产品范围残留

登录页仍出现“AI 建议，人来确认”；日历仍提及学生和家庭；设置模型仍保留隐藏 AI 通知键。当前产品边界明确不包含学生/家庭生命周期和已启用 AI 工作台。

验收要求：删除残留；智能建议正式启用时必须显示证据、规则版本、置信度和人工确认。

## 测试与部署缺口

- 现有 15 项 Node 测试主要检查源码字符串，不是运行时行为测试。
- 缺少数据库越权负向测试、API 认证契约、状态机、网络失败、键盘/焦点、视觉回归、Worker 崩溃重试和幂等测试。
- 本轮审计时 CRM Supabase 容器已退出，未重跑 pgTAP。
- 当前会话没有可用浏览器实例，未执行桌面/平板/375px 视觉验收。
- Sites 没有生产项目或部署，生产环境变量和调度未验证。

## 建议功能

### 优先运营能力

1. 权限解释器：解释可见范围、角色、owner/team 和 MFA 门槛。
2. 工作流滞留看板：审批、退款、导出、日历、通知和导入的 SLA、积压与重试。
3. 客户 360 活动记录器：统一记录电话、邮件、拜访、备注和下一步行动。
4. 续约 Playbook：90/60/30/14/7 天阶段、负责人、健康度、下一步行动和 SLA。
5. 商机阶段守卫：缺少下一步行动、关单原因或必要证据时禁止推进。
6. 重复数据合并预览：字段级差异、主记录选择、影响范围、冲突和审计。
7. 导入 Dry Run：应用前显示新增、更新、合并、跳过、冲突和错误。

### 集成与分析

1. Microsoft 365 / Google Calendar / 邮件集成状态、增量游标和重授权框架。
2. 电子签与财务系统签名 Webhook、幂等收件箱、失败重放和审计。
3. 产品包、选配项、折扣边界和报价策略基础模型。
4. 汇率锁定快照和本位币报表口径。
5. 续约、留存、队列 SLA、预测准确率和规则建议采纳率分析。

### 可解释的 Next Best Action
