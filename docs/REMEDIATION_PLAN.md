# Lumina Education CRM 整改计划与完成记录

计划日期：2026-07-17
完成版本：0.7.0
原则：保持 Vinext/Next + Supabase；问题多大改多大；不以重写掩盖局部缺陷。

## A. 产品边界与认证 — 已完成

- 删除所有外部用户注册页面、API、公开用户名检查和验证流程。
- 关闭 Supabase Auth 匿名 signup；仅管理员后台创建员工，管理员仅由超级管理员创建。
- 删除演示登录旁路、固定账号和跨项目注册验证配置。
- 初始化管理员同步到真实 Auth，随后移除 `ADMIN_PASSWORD`。
- 所有人登录强制 Turnstile，服务端 Siteverify；失败在验证器旁提示并自动刷新。
- 创建账号生成随机临时密码，通过受控邮件端点一次性发送；发送失败回滚账号。
- 首次登录强制改密；超级管理员和管理员随后强制 TOTP MFA/AAL2，AAL1 被页面、API 和 RLS 拦截。
- 登录/找回密码/改密/MFA 错误留在对应表单；URL 不携带错误和凭据。

## B. 权限与审批 — 已完成

- 六类员工角色来自 workspace membership，账号角色/等级/状态不可自改。
- 资源 RLS 按管理员、销售层级、负责人和显式协作者限制。
- 合同签署、价格、付款、退款、业绩和审批使用受控 RPC。
- 普通管理员/超级管理员两级审批，maker/checker、自审阻止、唯一待审、过期和执行状态已完成。
- 合同签署和业绩分配审批推进真实状态；导出审批创建唯一后台任务。
- 审计快照对联系人联系方式和付款引用脱敏。

## C. 真实数据与长列表 — 已完成

- 核心页面不再使用 fixture 回退；失败显示就地错误。
- 工作台、通知、员工、客户组织、联系人、任务、商机、合同、产品、日历和报告接入真实数据。
- CRM 表格、员工、合同、商机、审批、安全审计、通知和导出采用服务端搜索/分页。
- 长下拉使用通用可搜索选择器；组织/联系人采用远程搜索。
- 固定角标和固定全局搜索结果已删除。
- 未形成完整闭环的非核心模块明确停用，不返回假成功。

## D. 核心业务 — 已完成

- 销售层级、经理目标分配、直接/协助贡献、防重复与审批锁定。
- 月/季/年销售目标、实际、加权预测、漏斗、成员和关系覆盖分析。
- 四级关系目标证据与 4×4 关系/关单推进建议。
- 五个默认产品、自定义产品、多币种价格版本和未来生效。
- 合同草稿、版本、签署、续约父合同和续约提醒。
- 双月日历、用户时区、预约、关联记录、提醒和通知偏好。
- 已确认付款驱动的月/季/年消费报告；不同币种不直接相加。
- 审批导出 CSV、私有桶、签名下载、公式注入防护、失败诊断和过期清理。

## E. 国际化与 UI/UX — 已完成

- 中英文目录键一致；删除英文到中文的隐式回退。
- 当前语言决定组织、产品、任务等业务内容；人员姓名固定显示中文/英文。
- 页头和登录页均有语言切换，语言选择持久化。
- 登录、设置、合同、产品、商机、审批等错误显示在发生位置附近。
- 通用搜索、分页、进度、状态、消息、选择器和布局组件统一。
- 44px 触控目标、移动端卡片化、溢出约束、ARIA 进度/组合框/列表框已补齐。
- 关键弹窗支持 Escape 和背景滚动锁定；仍需 Chromium 可视验收确认所有焦点恢复细节。

## F. 安全、测试、版本与部署 — 基本完成

- 完成安全响应头、来源校验、登录限流、Turnstile、AAL2 门禁和健康检查。
- TypeScript、Lint、生产构建、14 项 Node 测试和 15 项 pgTAP 权限测试已通过。
- 真实管理员 Turnstile 登录、首次改密、TOTP/AAL2、员工创建邮件、专用 3200 端口和核心认证 API 已通过 HTTP 验收。
- 版本升级为 0.7.0；审计、状态和部署文档已更新。
- Chromium 1228 因当前运行时没有浏览器实例而未执行，是唯一未能在本环境完成的验收门禁；发布前必须补跑。

## 发布顺序

1. 备份远程 Supabase，并在暂存环境应用迁移 001–020。
2. 关闭公开 signup，配置生产域名、Turnstile、账号创建邮件、TOTP MFA 和服务器密钥。
3. 运行管理员初始化并立即删除 `ADMIN_PASSWORD`。
4. 部署构建，调度 reminder/outbox/export workers。
5. 执行 Chromium 1228 桌面与移动端验收、maker/checker 和导出下载演练。
6. 私有发布，观察健康、错误、慢查询、失败任务和提醒送达。

## G. 审计建议后续路线图（已纳入，不等于已上线）

### P1：补齐关键运营闭环

1. **客户 360 时间线**：统一展示组织、联系人、商机、任务、活动、预约、合同、付款、关系证据和审批事件；复用当前权限范围、服务端分页和审计，不另建影子客户表。
2. **联系人同意与沟通偏好**：增加渠道授权、用途、来源、取得/撤回时间、保留期限和勿扰规则；营销触达与导出必须检查同意状态。
3. **报价—合同—收款—退款—对账状态机**：在现有合同/付款基础上加入报价版本、折扣审批、应收计划、部分付款、退款凭证和对账差异，保持金额口径可追溯。
4. **数据质量、导入与查重闭环**：启用前完成字段映射、预检、重复候选、人工决策、幂等小批次、逐行错误和批次回滚；现有停用页面不能先行开放按钮。
5. **日历外部送达**：在现有双月日历/提醒之上补充真实邮件邀请、变更/取消通知、失败重试和送达状态，不继续使用“已发送”假反馈。

验收门槛：数据模型、RLS、审计、回滚、服务端搜索/分页、中英文、移动端和 Chromium 1228 全部通过后才启用入口。

### P2：财务与生态集成

1. **多币种归一化**：维护汇率来源、锁定时点和本位币快照；报表同时保留原币，禁止按实时汇率回写历史业绩。
2. **Microsoft 365 / Google Calendar 与邮件**：采用 outbox、幂等键、增量游标和断线重授权，明确 CRM 与外部系统的主数据归属。
3. **电子签与财务系统**：合同签署状态、发票、收款和退款通过签名 webhook 接入；所有回调验签、去重、审计和可重放。
4. **产品组合与报价策略**：支持产品包、可选项、折扣边界、审批矩阵和有效期；不允许销售绕过价格版本直接改历史金额。

### P3：可解释的智能建议

1. 先使用结构化规则生成 Next Best Action，再允许模型补充建议；每条建议显示证据、规则/模型版本、置信度、适用阶段和不适用原因。
2. 建议只能转为草稿任务，不能自动发消息、改阶段、承诺折扣或提交付款；敏感客户关系内容遵循最小化记录。
3. 建立采纳/驳回反馈、质量抽检、漂移监控和停用开关后，才从当前“未启用”状态开放 AI 工作台。

### 产品边界决策门

学生、家庭和升学专属生命周期不是当前 CRM 的既定核心范围，也绝不是登录用户体系。后续若无教育产品运营公司的明确业务案例，应删除这些停用入口，而不是因为旧原型存在就继续开发；任何决定都不得重新引入家长注册、导师、督导或 Little Spark 模型。
