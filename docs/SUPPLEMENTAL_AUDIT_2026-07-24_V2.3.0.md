# Lumina Education CRM v2.3.0 补充遗漏与基础 UX 审计

- 审计日期：2026-07-24
- 基线提交：`64b1f44 feat: release v2.3.0 audit remediation`
- 输入：`REMEDIATION_PLAN_2026-07-23_V2.3.0.md`、
  `FINAL_REAUDIT_2026-07-23_V2.3.0.md`
- 范围：既有计划完成度、设置与 MFA 业务状态、基础表单 UX、浏览器验收覆盖
- 当前状态：确认 5 项补充问题和 1 项新增部署要求；未发现既有计划中完全未实现的条目

## 结论摘要

既有 v2.3.0 计划中的供应链、会话、缓存、环境、MFA 说明、密码策略、头像类型/体积、
排版与 QA 清理均有对应实现和通过证据。补充审计发现的问题不是原计划条目被漏做，而是
自动化只验证页面“可渲染”时未覆盖到的交互状态和服务端不变量。

## 确认问题

### P1：安全通知可以被完全关闭

- UI 文案明确说明安全通知不可完全关闭。
- 两个渠道开关仍可同时关闭，API 也接受并持久化该状态。
- 影响：用户看到的组织安全策略与实际投递偏好矛盾；直接请求还能绕过未来的前端限制。
- 修复要求：UI 始终保留至少一个渠道，API 拒绝两个渠道同时关闭，读取历史异常值时安全
  归一化为站内通知开启。

### P1：MFA 未完成注册会留下未验证因子

- 设置页和首次 MFA 页面先创建因子，再单独创建 challenge。
- challenge 失败、刷新或重新开始时，之前的未验证 TOTP 因子不会被清理。
- 必须启用 MFA 的角色又被现有 `unenroll` 规则禁止删除任何因子，包括未验证因子。
- 影响：重复配置可能积累残留因子或触发身份提供商的重复/数量限制，用户只会看到笼统失败。
- 修复要求：注册前清理当前用户的未验证 TOTP 因子；服务端一次返回 factor 和 challenge；
  challenge 创建失败时回滚新因子；强制 MFA 角色只禁止删除已验证因子；设置页提供取消本次
  配置入口。

### P2：无效头像选择仍显示已取消的旧预览

- 用户先选择有效图片，再选择超类型或超 5 MB 图片时，待上传文件会被清空，但旧 blob
  预览仍留在界面。
- 影响：界面暗示旧预览仍会保存，实际提交时不会上传，属于明显的低级反馈错误。
- 修复要求：无效选择同时撤销待上传文件和临时预览，恢复已保存头像或姓名缩写。

### P2：浏览器验收遗漏恢复页和多项设置/管理页

- 现有 43 项覆盖登录、主要业务页、两个设置页和单个管理页。
- 未覆盖找回密码、重设密码、账户、通知、隐私、管理员用户和安全页；通知不变量也没有交互
  验证。
- 影响：这些页面的溢出、可访问名称、标题层级、低对比度和基础交互回归可能在 release gate
  中漏过。
- 修复要求：继续使用固定的 `ms-playwright/chromium-1228`，补齐桌面/移动页面检查，并验证
  安全通知最后一个渠道不能关闭。

### P2：管理员安全页存在 10px 运营文字

- 扩大后的 Chromium 1228 矩阵在 `/admin/security` 桌面与移动视口发现同一问题：
  指标来源说明和审计动作标签共 21 处为 10px。
- 页面没有溢出、未命名控件、低对比度或业务错误，但该字号明显低于本项目 12px 可读性基线。
- 修复要求：把安全说明和审计动作标签纳入全局 release typography floor，并同时复验
  1440px 与 375px。

### 新增要求：一键生产部署必须有硬超时

- 生产环境采用 Git 拉取、专用服务器、systemd 和原子 release 目录，但仓库原先只有手工
  步骤，没有可执行的一键部署入口。
- 用户明确要求所有任务都有持续时间上限，不能因网络、依赖、构建、迁移或服务启动无限卡住。
- 实现要求：一行命令完成 fast-forward pull、独立构建、迁移、原子切换、systemd 重启和
  健康检查；总流程与每一步均强制超时；切换后失败自动回滚应用版本。

## 非问题与边界

- MFA 推荐终端说明已经覆盖 Microsoft Authenticator、Google Authenticator，以及组织集中
  管理时的 1Password，并包含自动校时、设备锁和 QR/secret 防泄漏说明。
- 管理员安全页使用真实仓储数据，不是静态演示数字。
- 生产密钥、托管邮件模板、备份恢复演练和目标环境迁移仍属于部署准备，不是本次代码缺陷。
