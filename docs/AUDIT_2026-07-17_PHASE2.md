# Lumina Education CRM 第二阶段完整性审计

审计日期：2026-07-17
基线版本：0.7.0
范围：客户 360、联系人同意、报价至对账、数据质量与导入、日历外部送达，以及上一轮 UI/UX 收尾。

## 结论

0.7.0 的员工认证、权限、销售、合同、日历、报表和国际化核心可继续复用，但五项新增需求尚未形成完整业务闭环。现有代码没有被中途指令破坏；主要缺口来自此前明确延后的路线图，而不是半截实现。本阶段采用增量迁移和现有 Supabase 权限边界，不重写认证、导航或核心表。

## 审计发现

### 1. 客户 360

- 组织、联系人、商机、任务、活动、预约、合同、付款、关系证据和审批事件已分散存储，但没有统一时间线。
- 必须从原始业务表按访问范围实时组合，不能复制到影子客户表。
- 需要稳定游标/服务端分页、类型筛选、统一事件结构和来源链接。

### 2. 联系人同意与沟通偏好

- 联系人目前只有邮箱和电话，没有渠道×用途授权、来源、取得/撤回时间、保留期限或勿扰规则。
- 营销邮件和导出缺少服务端同意门禁；仅靠界面隐藏不足以满足要求。
- 服务/交易通知和营销触达必须分开判断，撤回记录必须保留审计。

### 3. 报价、合同、收退款与对账

- 现有合同版本和付款确认可追溯，但没有报价版本、折扣审批和应收计划。
- 付款状态只有整笔确认/整笔退款，不能正确表达部分付款、部分退款和多次收款。
- 退款缺少独立凭证与审批状态，对账差异没有可处理的业务对象。

### 4. 数据质量、导入与查重

- 新建联系人/组织已有即时重复检查，但导入、重复候选、人工决策、批次回滚和逐行错误页面仍停用。
- 旧页面文案描述了完整能力，实际入口却不可操作；本阶段必须以真实持久化和幂等处理替换占位。
- 合并属于高风险操作，必须保存前后快照、冲突检测和重挂关系结果。

### 5. 日历外部送达

- 双月日历、预约、提醒与通用 outbox 已存在，但参与人邮件邀请仍显示“后端接入后发送”。
- 缺少邀请版本、变更/取消事件、逐收件人状态、重试时间和最终失败原因。
- 页面不得在外部提供商确认前显示“已发送”。

### 6. 上一轮 UI/UX 收尾

- 停用模块仍出现在主导航，增加认知负担。
- 管理门户问候姓名和 MFA 告警标题存在固定示例文本。
- 国际化目录保留未使用的邀请、演示任务和 MFA 占位键。
- Chromium 1228 的桌面、375px、键盘焦点和溢出验收仍是发布门禁。

## 架构风险控制

- 所有新表带 `workspace_id`、RLS、审计触发器和必要索引；服务端继续使用现有用户令牌，不新增旁路密钥访问。
- 时间线通过 SQL 函数联合原表生成，避免双写和数据漂移。
- 金额事件使用不可变版本和独立流水，不通过覆盖历史金额表达退款。
- 导入以内容幂等键、批次状态、逐行结果和补偿回滚实现；不承诺无法回滚的跨批次合并。
- 同意检查在数据库函数/API/导出或送达 worker 三层中至少两层执行，前端不是安全边界。
- 外部邮件采用 outbox、幂等键、指数退避和明确状态；日志不得记录敏感正文或临时密码。

## 整改复核

复核日期：2026-07-17

五项缺口均已按 `DEVELOPMENT_PLAN_V0.8.0.md` 实现并启用真实入口。实现保持增量边界：没有重写认证、没有新建影子客户表、没有引入第二套权限系统，也没有恢复家长注册、导师、督导或 Little Spark 相关模型。旧学生、家庭、升学、独立线索和 AI 占位路由及其未使用语言内容已删除，而不是仅从导航隐藏。

复核执行了类型检查、Lint、生产构建、Node 架构契约、pgTAP 权限结构和本地认证业务冒烟。冒烟曾发现两个上线阻断项：报价函数的 `version` 字段歧义，以及导入行无重复候选时覆盖 JSON 空数组默认值；现已分别通过向前迁移 `023`、`024` 修复，并在修复后整链路通过。

仍未关闭的不是开发缺口，而是环境性发布门禁：必须在可用的 `ms-playwright/chromium-1228` 中完成 1440px、1024px、375px、键盘焦点、弹层恢复、就地错误和横向溢出验收。未完成该验证前，不应把 v0.8.0 宣布为已完成生产视觉验收。
