# Lumina CRM v1.1.0 修复与产品完善计划

制定日期：2026-07-18
输入：`AUDIT_2026-07-18_POST_RELEASE.md`

## 目标

在不削弱现有安全边界的前提下，关闭 v1.0.0 审计中的全部 P0/P1/P2 缺陷，并将六项产品建议整合到现有信息架构中，最终形成可构建、可测试、可部署、可回滚且业务口径明确的 v1.1.0。

## 实施原则

- 数据库与服务端是业务规则和统计口径的最终来源。
- 客户端和接口共享输入契约，避免界面允许、接口拒绝。
- 所有业务总数由独立聚合返回，不能从截断列表推导。
- 所有请求统一具备超时、刷新、Abort、错误码和 request ID。
- 所有模态交互使用统一可访问组件。
- 用户查看时区与工作区财务报表时区职责明确。
- 烟测必须可重复执行并清理自身数据。
- 新功能优先复用现有数据质量、自动化和运行中心，不制造重复入口。

## 阶段一：核心销售流程

- 提取商机创建和阶段转换共享 Schema。
- 新建商机表单补齐必填状态、客户端校验和字段错误。
- 引入阶段转换引导器：
  - 活跃阶段：预计成交日期、中英文下一步行动。
  - Won：成交凭证。
  - Lost：失败原因。
- 提交期间禁用重复操作，失败时保留输入并显示业务错误码。
- 更新成功后刷新列表、漏斗和分页总数。
- 为创建及六种阶段转换增加可执行契约测试。

验收：用户可以完成 Discovery、Evaluation、Hesitation、Payment、Won、Lost 的合法转换；非法转换在提交前被阻止。

## 阶段二：财务数据正确性与风险中心

- 将财务合同、应收、付款、退款和对账改为独立分页查询。
- 新增服务端精确风险聚合：逾期应收、未结清应收、待处理退款、对账异常。
- 删除把对账错误回退为空数组的行为。
- 财务页面增加风险中心、分类筛选、分页和显式错误状态。
- 所有金额聚合明确币种，禁止把不同币种直接相加。

验收：列表可以访问完整数据；风险指标不受当前页影响；服务故障不会显示为零风险。

## 阶段三：CRM 录入与指标

- 学校表单：中英文名称、城市、课程体系、联系人信息。
- 联系人表单：中英文名称、职位、邮箱、电话和所属机构。
- 任务表单：中英文标题、截止日期、优先级、负责人和关联对象。
- 数据查重结果在任一相关字段变化后自动失效。
- 新增资源统计接口，返回精确总数、需关注数和平均完整度。
- 列表默认每页 10 条，支持 10/25/50 条。
- 修复非法页码和筛选后越界页码。

验收：三类记录能录入业务关键字段；概览指标来自全量筛选结果。

## 阶段四：平台一致性

- 建立共享日期格式化能力，并在用户偏好上下文中提供时区和日期格式。
- 修复仪表盘跨日/跨月口径，明确工作区报表时区。
- 迁移直接 `fetch` 到统一 API 客户端。
- 统一显式错误、空数据、重试和 request ID 展示。
- 修复全部未定义 CSS Token，增加静态 Token 测试。
- 将自定义抽屉迁移到共享可访问抽屉。
- 完成移动导航、通知和用户菜单的键盘语义。

验收：设置时区后所有主要页面一致；核心组件键盘可达；没有未定义 Token 和无超时请求。

## 阶段五：环境、测试和发布

- 本地环境脚本由覆盖改为合并，并生成当前版本全部本地必需变量。
- 添加环境变量 Schema 和就绪检查，生产环境对关键密钥 fail closed。
- 修复烟测心跳清理及数据库测试顺序依赖。
- 增加 UI/API 业务契约、分页、统计、Token 和可访问性静态测试。
- 执行 TypeScript、ESLint、生产构建、Node 测试、npm audit、数据库 lint、数据库测试和全套烟测。

验收：README 推荐顺序可连续重复执行两次并全部通过。

## 阶段六：产品能力整合

### 商机阶段引导器

由阶段转换弹窗承担，不再增加重复页面。

### 财务风险中心

整合进财务页面顶部，并提供跳转到对应风险列表的筛选入口。

### 数据完整度工作台

复用数据质量页面，增加资源缺失字段分类、目标记录链接、负责人和处理状态。

### 销售自动化

复用现有下一步行动与规则体系，增加“长期未跟进、预计成交临近、续约临近、付款逾期”四类规则状态和运行结果。

### 保存视图与团队仪表盘

- 个人保存视图保存于设备本地，不承载敏感业务数据。
- 保存查询、状态、排序、方向和每页条数。
- 提供恢复默认和删除入口。
- 团队级共享视图作为后续数据库能力保留扩展点。

### 发布与运行控制台

复用管理员运行中心，集中展示 API、数据库、Worker、Webhook、集成、环境配置和最近质量门禁状态，并给出可执行修复建议。

## 最终反查

完成开发后重新逐条核对本计划和审计报告：

1. 每个发现必须映射到代码、测试或明确的外部依赖。
2. 不允许以隐藏错误、截断列表或样例数据代替真实实现。
3. 不允许把“已有页面入口”误报为“功能完成”。
4. 未通过浏览器、数据库或生产环境验证的项目必须明确标注，不能宣称已完成。
5. 对发现的遗漏继续修复，直到没有可在当前仓库内完成的遗留项。

## 发布标准

- 所有当前仓库内可执行的 P0/P1/P2 修复完成。
- 六项产品建议均具备可用入口和真实业务逻辑，或清楚标注依赖外部配置的运行状态。
- 全部自动化质量门禁通过。
- 完成浏览器桌面、平板、移动端和键盘验收。
- 保存精确源码版本并部署为私有 Sites 版本。

## 执行结果

截至 2026-07-18，阶段一至阶段六的全部仓库内工作均已完成。最终复审发现的付款时间时区转换和联系人/任务选择字段前置校验也已补齐。自动化门禁结果为 Node 20/20、pgTAP 132/132、数据库 lint 0 错误、依赖漏洞 0，五组 HTTP/业务烟测全部通过。

浏览器实例、生产运行时密钥、正式 Supabase、外部 provider 和六个 Worker 调度不属于仓库可生成的状态。当前 Sites 项目仍无生产环境变量；依据本计划 fail-closed 原则，不部署一个已知 readiness 503 的生产版本。详细关闭映射见 `FINAL_REAUDIT_2026-07-18_V1.1.0.md`。
