# Lumina Education CRM v2.4.0 完整补充审计

- 审计日期：2026-07-24
- 审计基线：`8d42df6 fix: close supplemental v2.3.0 audit gaps`
- 范围：代码与架构、业务不变量、安全、UI/UX、可访问性、测试与发布流程
- 目标版本：v2.4.0

## 结论摘要

v2.3.0 的类型、Lint、生产构建和依赖审计基线均通过，既有通知、MFA、头像预览、
排版和部署修复也确实存在。本轮继续审计后确认 6 类此前未关闭的问题。最严重的问题不是
页面能否渲染，而是测试和网络请求缺少统一的“总时限 + 无输出时限”：发布门可以在某个
子进程静默卡住二三十分钟，应用也可能在 Supabase/Auth/Storage 半断开时无限等待。

## 已确认问题

### P0：发布门没有子阶段超时、无输出超时或总时限

- `scripts/release-gate.mjs` 的通用 `run()` 只等待子进程退出。
- typecheck、Lint、构建、npm audit、pgTAP、smoke 和 Chromium 任一阶段卡死时，没有终止
  进程树、阶段进度、最后输出时间或总预算。
- GitHub Actions 只有 20 分钟 job 上限，不能帮助本地执行，也无法快速识别具体卡住阶段。
- 现有 Chromium 单动作 12 秒只覆盖 Playwright 动作，不能防止脚本、清理或子进程整体卡住。

影响：失败反馈极慢，容易残留后台进程，并反复浪费 20–30 分钟。

### P1：核心上游请求没有统一硬超时

- `supabaseRequest()` 和 `supabaseAdminRequest()` 是绝大多数仓储调用的单一入口，但没有
  `AbortSignal` 超时。
- 登录、设备 OTP、session refresh、密码恢复、头像上传、当前密码验证和浏览器端认证表单
  仍存在裸 `fetch()`。
- 上游连接半开时，Server Component、API 和表单都可能永久 pending。

影响：加载器不结束、请求占用资源、用户无法判断是慢还是失败，readiness 也可能被拖住。

### P1：API 错误规范化仍可能回传底层消息

- 多个路由把 `SupabaseRequestError.message` 放到响应。
- `normalizeErrorResponse()` 又保留顶层 `message` 并复制到结构化错误。

影响：约束名、RPC 文本或底层服务细节可能暴露给浏览器；前端实际只依赖稳定错误码。

### P1：Chromium 1228 矩阵仍遗漏可导航业务页

遗漏至少包括：

- `/reports`、`/reports/exports`、`/reports/marketing`
- `/analytics/consumption`、`/sales/allocation`
- `/duplicates`、`/data-quality`、`/help`
- `/admin`、`/admin/approvals`

影响：这些页面的真实数据加载、标题、溢出、低于 12px 文字、对比度和移动布局不在发布门内。

### P2：头像只相信浏览器提供的 MIME

- 服务端检查 MIME 和 5 MB 上限，但没有核对 PNG/JPEG/WebP 文件签名。
- 用户更换扩展名时，旧私有 Storage 对象不会清理。

影响：可保存伪装图片或残留不可达对象，造成失败预览和存储垃圾。

### P2：找回密码与设置新密码共用同一页面标题

`/forgot-password` 和 `/reset-password` 都使用 `meta.resetPassword`，浏览器标题无法区分
“请求恢复邮件”和“设置新密码”两个不同阶段。

## 已验证的非问题

- 当前工作树在审计开始时干净，分支相对 `origin/main` ahead 2。
- TypeScript、ESLint 和生产构建通过。
- `npm audit` 为 0 vulnerabilities。
- 本地 Supabase 运行正常，精确 `ms-playwright/chromium-1228` 可用。
- 所有普通状态变更 API 均有同源检查；外部 webhook 使用独立 HMAC、时间窗和幂等逻辑。
- v2.3.0 的安全通知不变量、MFA 未验证因子清理和取消流程已落地。

## 本轮功能性改进建议

本轮直接实现以下高价值能力，而不是继续扩大业务域：

1. 可复用的有界子进程执行器：阶段时限、静默时限、心跳、进程树终止和明确退出原因。
2. 所有标准质量入口默认有界，发布门再提供第二层总预算。
3. 上游请求统一超时并返回稳定 `UPSTREAM_TIMEOUT`/服务不可用语义。
4. 浏览器 QA 扩展为完整导航矩阵，并保存精确运行证据。
5. 头像真实格式验证与旧对象回收。

## 执行中发现的补充问题

分阶段执行扩大后的 Chromium 矩阵及最终遗漏反查又发现并关闭了以下问题：

- 消耗分析、帮助和管理员卡片存在低于 12px 的正文或对比度不足。
- 销售分配进度标签被通用 `article > span` 样式覆盖，出现 3.92 的低对比度。
- 375px 管理治理摘要溢出；425px 管理状态徽章把页面撑出视口。
- 有界执行器从 `npm run` 外直接调用时错误依赖 `npm_execpath`，导致独立审计阶段快速失败。
- 发布门默认使用 `127.0.0.1`，与生产模式 Secure Cookie 的本地 `localhost` 验证语义不一致。
- 最终检索发现 `lib/auth.ts` 的用户资料、membership 与当前用户请求未走统一网络超时。

这些问题均已在 v2.4.0 代码中修复，并纳入 31 条 Node 契约或真实 Chromium/鉴权 smoke。
