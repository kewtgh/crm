## 发现

### SEC-01 · 高危供应链漏洞阻断发布（P0）

`npm audit --audit-level=moderate` 在基线锁文件中报告 7 个 high：

- Next.js 16.2.10 受中间件/Proxy 授权绕过等公告影响，官方修复版本为 16.2.11。
- `sharp` 0.34.5 继承 libvips 高危问题，修复线为 0.35.x。
- `brace-expansion` 1.1.14/5.0.6 和 `fast-uri` 3.1.2 存在 DoS/URI 混淆问题。

发布门禁在依赖审计阶段正确失败。应升级 Next.js 与匹配的 ESLint 配置，使用兼容 override
固定 `sharp` 0.35.3，并采用非破坏性 audit fix 更新其余传递依赖；禁止使用会把核心框架或
Cloudflare 插件降级的 `npm audit fix --force`。

最终发布门禁又捕获了 2026-07-23 才进入审计数据库的 PostCSS 公告
`GHSA-6g55-p6wh-862q`：受影响版本为 `<=8.5.11`，官方修复线为 8.5.12。由于本项目此前
override 为 8.5.10，整改范围追加固定 PostCSS 8.5.14，并重新从头执行门禁。

### AUTH-01 · “记住我”会话期限在续期后被扩大（P0）

登录、设备验证和 MFA 完成路径在未勾选“记住我”时创建 session refresh cookie；但
`/api/auth/refresh` 无条件给旋转后的 refresh token 设置 30 天 `maxAge`。因此一次正常的
访问令牌续期会把非持久会话升级为持久会话。访问 Cookie 在未勾选时也不应携带 `maxAge`。

应保存一个 HttpOnly、SameSite=Lax 的服务端持久性标记，并让登录、设备验证、MFA、续期、
登出和密码变更共享同一 Cookie 语义。未勾选时 access、refresh 和标记均只存在于浏览器会话；
勾选时刷新令牌和标记最多保存 30 天。

### API-01 · 敏感 JSON 未统一声明禁止缓存（P1）

大部分 API 经 `apiRoute` 统一处理请求 ID 和错误形状，但成功与失败响应没有统一
`Cache-Control: no-store`。虽然带 Cookie 的响应通常不会被共享缓存主动缓存，CRM 中包含人员、
财务、隐私和认证数据，不应依赖启发式缓存行为。

应由 API 包装器统一添加 `no-store`，并补齐未使用包装器的认证刷新/登出路径；已有更明确
缓存策略的响应应保持其显式策略。

### OPS-01 · 本地环境生成器与运行时契约不完全一致（P1）

`scripts/configure-local-environment.ps1` 漏掉 `WEBHOOK_PAYMENT_SECRET`，也没有显式写入
`WEBHOOKS_ENABLED`、`INTEGRATION_SYNC_ENABLED`、`AI_PROVIDER_ENABLED` 和
`EXPORT_MAX_ROWS`。默认关闭逻辑能避免误启用，但生成的环境文件无法完整表达 `.env.example`
和运行时检查器的契约，启用支付 Webhook 时会形成不必要的配置缺口。

应补齐支付 secret、显式关闭可选外部能力、写入导出硬上限，并保留既有用户值。

### QA-01 · 固定浏览器脚本依赖易失的 npx 临时目录（P1）

Chromium 1228 的可执行文件存在且版本固定，但 device-auth 与 UI QA 把 `playwright-core`
写死到一次性 npx cache hash。正常的 `npm install` 清理旧 cache 后，门禁会在加载控制库时
失败，容易被误判成浏览器不可用。

应把 `playwright-core` 固定为只提供协议控制能力的开发依赖；仍只启动
`ms-playwright/chromium-1228`，禁止下载或替换浏览器。QA 报告同时记录控制库版本、
浏览器版本和 executable。

### UX-01 · MFA 配置缺少终端选择与安全说明（P1）

现有界面只提示“使用身份验证器扫描二维码”，没有说明这是 6 位 TOTP、Microsoft
Authenticator 中应选择 Other account、推荐平台、设备时间要求，以及二维码/手动 secret
不得分享。员工容易误以为必须使用某一厂商的推送验证，或把 secret 截图发给支持人员。

应在首次强制配置和账户安全设置中复用同一中英文指南。推荐组织批准的 Microsoft
Authenticator 或 Google Authenticator（iOS/Android）；若组织已经统一管理 1Password，
可使用其内置一次性密码功能。Lumina 只接收标准 6 位 TOTP，不发送厂商 push approval。

### UX-02 · MFA 操作错误显示在密码表单中（P1）

账户安全页用同一组 `error/success` 状态处理 MFA 和修改密码，但消息节点位于密码表单底部。
MFA 加载、启用或停用失败会在不相关的密码字段之后出现，属于高混淆、低实现成本的基础 UX
问题。应拆分 MFA 与密码状态，并在各自操作附近显示反馈。

### AUTH-02 · 三条密码更新路径策略不一致（P0）

首次登录要求至少 12 位并同时包含大写、小写和数字；账户设置仅要求 10 位，找回密码也只在
客户端检查 10 位、大写和数字。员工可通过不同入口绕过更强策略。

应导出一个共享密码 schema，首次登录与账户设置服务端都使用同一策略；找回密码客户端使用
同一 schema，并让所有输入的 `minLength/maxLength` 和帮助文案一致。

### UX-03 · 头像大小说明与服务端限制不一致（P2）

设置页写“最大 3 MB”，API 接受 5 MB，文件选择后也不做即时类型/大小反馈。应统一为 5 MB，
并在本地预览和上传前进行同一 MIME/大小校验，把错误放在头像控件旁。

### UX-04 · 核心运营页面残留 8–11px 文本与灰色状态对比度不足（P1）

固定 Chromium 1228 的真实渲染检查发现，日历日期、合同摘要、关系目标、设置说明和集成配置
仍有 8–11px 的可见文字；“未连接”灰色状态在浅灰底上的对比度为 4.16:1，低于普通文本
4.5:1 门槛。这属于此前样式表后段规则覆盖早期 12px 可读性修复的基础级联问题。

应在所有组件规则之后建立针对运营标签的 12px typography floor，并加深灰色状态文字；同时
保留移动端卡片和日历布局，不以隐藏内容规避检查。

### QA-02 · 浏览器 QA 清理遗漏自动化事件且报告写入时机错误（P1）

工作流交互会为临时经理身份生成 `automation_events`，清理 SQL 未删除这些事件，导致 Auth
临时身份因外键无法删除。脚本还在追加 cleanup mismatch 之前写报告，报告内容会少记最后一项。
应补齐精确 actor 清理、让单个身份清理失败不阻断其余身份，并在完整记录 cleanup 错误后落盘。

## 架构与 UI/UX 复核

- 服务端页面和 API 都执行能力/角色检查，关键写操作有同源校验；Webhook 使用时间窗、
  原始 payload digest、HMAC 和常量时间比较。
- PostgreSQL 业务操作以 RPC/事务和 RLS 为主，当前 433 条断言未发现跨 workspace、越权或
  状态机回归。
- 全局搜索支持 Ctrl/⌘K、键盘选择和移动抽屉；Drawer、菜单、通知弹层具备焦点进入、循环、
  Escape 关闭和焦点恢复；页面具备 skip link、可见焦点和 reduced-motion。
- 列表已使用统一 10/20/50 分页契约；增长型数据未发现新的客户端全量加载。
- `app-shell.tsx`、`pipeline-page.tsx`、`imports-page.tsx` 和数个 repository 已接近或超过
  400 行，是维护性热点，但当前职责边界仍可辨识。安全补丁中拆分会增加回归面，建议在后续
  独立重构版本中按“导航/搜索/弹层”和“schema/query/mapper”拆分，并保持公开接口不变。
- 当前产品域已经覆盖客户 360、教育关系、线索/商机/合同/财务、隐私、自动化、导入导出和
  运维治理。本轮不建议再引入未经真实用户验证的大型功能；优先把会话策略、安全依赖、
  环境复现和固定浏览器门禁做成可信发布基线。
